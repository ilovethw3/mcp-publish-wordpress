# 需求文档：内容控制平台 (MCP) v1.1

## 1. 简介

本项目旨在创建一个名为“内容控制平台”(MCP) 的后端服务和一个配套的前端审核仪表盘。MCP的核心功能是接收由外部AI Agent自动生成的文章草稿，提供人工审核、编辑和批准的界面，并将最终批准的文章通过API发布到指定的WordPress网站。

该系统旨在实现内容生产流程的自动化，同时通过强制的人工审核环节来保证最终发布内容的质量。

## 2. 功能需求

### 2.1 接收文章草稿

- **用户故事**: 作为一个外部AI Agent，我希望能将生成的文章草稿安全地提交给MCP进行处理，以便后续的人工审核和发布。
- **需求**:
  1.1. MCP **必须** 提供一个公开的、安全的API端点，用于接收文章草稿。
     - **当** 接收到一个带有有效认证凭证的HTTP POST请求时，
     - **且** 请求体包含预定义格式的JSON数据，
     - **则** 系统 **应** 成功接收文章草稿，将其存入数据库，并设置初始状态为“待审核”(pending_review)。
  1.2. MCP **必须** 对接收的API请求进行认证。
     - **当** 请求缺少有效的认证凭证时，
     - **则** 系统 **应** 拒绝该请求并返回 `401 Unauthorized` 或 `403 Forbidden` 状态码。
  1.3. MCP **必须** 验证接收数据的格式。
     - **当** 接收到的JSON数据缺少必要字段或字段类型不正确时，
     - **则** 系统 **应** 拒绝该请求并返回 `422 Unprocessable Entity` 状态码。

### 2.2 用户认证与授权

- **用户故事**: 作为一个内容审核员，我希望能安全地登录MCP的审核仪表盘，以便只有授权人员才能访问和管理文章。
- **需求**:
  2.1. MCP **必须** 为审核仪表盘提供用户认证功能。
     - **当** 用户在登录页面提交了正确的用户名和密码时，
     - **则** 系统 **应** 返回一个有效的JWT (JSON Web Token)，用于后续的API请求认证。
  2.2. MCP **必须** 保护需要登录才能访问的API端点。
     - **当** 一个API请求缺少有效的JWT或JWT已过期时，
     - **则** 系统 **应** 拒绝该请求并返回 `401 Unauthorized` 状态码。
  2.3. 系统 **必须** 在数据库中安全地存储用户密码。
     - **在** 存储用户密码之前，
     - **则** 系统 **必须** 使用经过验证的强哈希算法（如Bcrypt）对其进行哈希处理。

### 2.3 文章审核与管理

- **用户故事**: 作为一个内容审核员，我希望能在仪表盘上查看、编辑、批准或拒绝待审核的文章，以便对内容质量进行把控。
- **需求**:
  3.1. 审核仪表盘 **必须** 显示所有处于“待审核”状态的文章列表。
     - **当** 我访问审核仪表盘时，
     - **则** 系统 **应** 调用API获取并展示一个包含文章标题和创建时间的列表。
  3.2. 审核仪表盘的文章列表 **应** 提供基础的搜索和排序功能。
     - **当** 我在列表页面时，
     - **则** 系统 **应** 允许我按文章标题进行搜索，并按创建时间进行排序。
  3.3. 审核仪表盘 **必须** 提供文章内容的编辑功能。
     - **当** 我从列表中选择一篇文章时，
     - **则** 系统 **应** 展示其完整的标题、内容、标签等信息，并允许我进行修改。
  3.4. 审核仪表盘 **必须** 提供批准文章的功能，并能处理发布过程中的各种状态。
     - **当** 我点击“批准并发布”按钮时，
     - **则** 系统 **应** 调用MCP后端API，触发发布流程，并将文章状态更新为“发布中”(publishing)。
     - **在** 文章发布失败后，状态 **应** 变为“发布失败”(publish_failed)，并且UI上 **必须** 提供一个“重试发布”的按钮。
  3.5. 审核仪表盘 **必须** 提供拒绝文章的功能。
     - **当** 我点击“拒绝”按钮时，
     - **则** 系统 **应** 调用MCP后端API，将文章状态更新为“已拒绝”(rejected)，并将其从待审核列表中移除。

### 2.4 发布到WordPress与结果追溯

- **用户故事**: 作为一个系统，我希望在文章被批准后，能自动将其发布到配置好的WordPress网站，并记录发布结果。
- **需求**:
  4.1. MCP **必须** 能够将文章内容发布到WordPress。
     - **当** 一篇文章的发布流程被触发时，
     - **则** 系统 **应** 使用预先配置的WordPress REST API凭证，调用相应的API接口创建一篇新的文章。
  4.2. MCP **必须** 正确处理发布的字段映射。
     - **当** 发布文章时，
     - **则** 系统 **应** 将MCP中的文章标题、HTML内容、标签、分类等字段正确地映射到WordPress文章的对应字段中。
  4.3. 系统 **必须** 实现发布结果的追溯。
     - **当** 文章成功发布到WordPress后，
     - **则** MCP **必须** 保存返回的WordPress文章ID和永久链接(Permalink)，并在审核仪表盘中提供一个可点击的链接，以便快速访问。

### 2.5 非功能性需求

- **需求**:
  5.1. **可观测性**: 系统 **必须** 对关键操作（如API调用、用户认证、数据库操作、发布流程）进行结构化日志记录，以便于问题排查和系统监控。
